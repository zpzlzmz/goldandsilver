# 대표 미팅용 요약: “왜 1인 개발이 어려운가” + “왜 기획/책임자가 필요한가” + “GoldSilver(금은세상) 프로젝트 개요”

## 1) 결론(대표님께 30초 요약)

- **AI로 프론트 구현 속도는 빨라질 수 있지만**, 제품이 “사업”이 되려면 **결정(우선순위/정의)과 책임(품질/보안/법적/운영)**이 필요합니다.
- **1인 개발은 ‘코드 작성’보다 ‘의사결정·품질·운영·리스크 관리’가 병목**이 되기 때문에, 출시가 가까워질수록 실패 확률과 비용이 급격히 커집니다.
- 따라서 **기획/제품 책임자(PO/PM)**와 최소한의 **기술 책임(Tech Lead/Backend/DevOps/QA)** 역할이 있어야 일정·품질·리스크를 통제할 수 있습니다.

---

## 2) 1인 개발이 “현실적으로” 어려운 이유 (코딩 말고, 나머지가 더 큼)

### A. 요구사항은 계속 변하고, “정의”가 없으면 개발이 영원히 반복됨
- “시세 비교”가 **어떤 업체를 비교**하는지, “AI 추천”이 **무엇을 근거로** 어떤 문구를 내야 하는지, “알림”이 **언제/어떻게** 울려야 하는지 등은 **코드가 아니라 정의의 문제**입니다.
- 정의가 없으면 개발자는 계속 “추측 구현 → 수정 요청 → 재구현” 루프에 갇히고, 그게 일정 폭발로 이어집니다.

### B. 금융/자산 관련 서비스는 보안·법무·신뢰가 핵심 병목
- “동작한다”는 것과 “출시해도 된다”는 다릅니다.
- 출시 수준에서 반드시 챙겨야 하는 것들:
  - **개인정보 처리/보관/파기**, **로그/모니터링**, **취약점 대응**, **권한 관리**
  - 약관/개인정보처리방침/마케팅 동의 등 **법무 산출물**
  - 데이터 출처·면책(투자 권유 아님) 등 **리스크 문구**
- 이걸 1인이 동시에 다 챙기면, 개발 시간은 자연히 “코딩 < 검증·문서·운영”이 됩니다.

### C. “버그 1개”가 매출/신뢰를 흔들기 때문에 QA와 릴리즈 프로세스가 필요
- 특히 거래/자산 계산/알림은 작은 오류도 CS 폭증으로 이어집니다.
- 1인이 개발+테스트+배포+운영까지 하면 **자기 검증 한계**가 있어서 결함이 쉽게 누적됩니다.

### D. 외부 API/플랫폼 의존은 ‘변경·장애’가 상수
- 공공데이터/카카오 같은 외부 API는
  - 장애/지연, 응답 포맷 변경, 쿼터 제한, 키 관리, 비용 이슈가 언제든 생깁니다.
- “연동했다”에서 끝이 아니라, **모니터링·대응·대체 경로(폴백)**를 운영해야 합니다.

### E. 출시 후 유지보수가 시작(실제 비용은 운영에서 발생)
- 사용자 증가 시 이슈는 늘고, 기능은 계속 추가됩니다.
- 1인은 운영이 시작되는 순간부터 **개발 속도가 급락**합니다(CS/장애/핫픽스/리그레션).

---

## 3) “AI가 프론트를 대체한다 해도” 기획/책임자가 반드시 필요한 이유

### A. AI는 “결정”을 대신하지 못함
- AI는 코드/문구를 생성하지만, 아래는 누군가 결정해야 합니다:
  - **무엇이 핵심 KPI인지**(재방문? 전환? 결제?)
  - **무엇을 먼저 만들지**(우선순위)
  - **성공 기준이 무엇인지**(Definition of Done)
  - **사용자에게 무엇을 약속할지**(정책/제한/면책)

### B. 책임 소재(리스크 오너십)는 AI가 질 수 없음
- “알림이 잘못 울려서 손실 봤다”, “가격 표기가 혼동된다”, “개인정보가 노출됐다”
  - 최종 책임은 **회사**가 집니다.
- 그래서 **제품 책임자(PO/PM)**와 **기술 책임자(Tech Lead)**가 필요합니다.

### C. 기획은 문서가 아니라 “의사결정의 연속”
- PRD/와이어프레임/정책/경계조건(예: 단위 변환, 세금, 매도 제한) 정의
- 사용자 흐름/예외 케이스/CS 시나리오 설계
- 경쟁사/가격 정책/차별점 정리

---

## 4) 최소 역할 구성(“사람 수”가 아니라 “역할”이 필요)

### 출시 전 최소한으로도 필요한 역할(겸직 가능)
- **PO/PM(제품 책임)**: 요구사항 정의, 우선순위, 일정/스코프 조정, 정책/문구/면책, CS/운영 기준
- **Tech Lead(기술 책임)**: 아키텍처 결정, 품질 기준, 성능/보안 가드레일, 코드 리뷰 기준
- **QA(품질 책임)**: 테스트 케이스, 회귀 테스트, 릴리즈 체크리스트
- **DevOps/운영**: 배포 파이프라인, 모니터링, 장애 대응 프로세스

> 1인 개발이 “불가능”이라기보다, 위 역할을 1명이 동시에 수행하면 **속도·품질·리스크가 통제 불가**가 되기 쉬워서 “사업 관점에서 비효율”이 됩니다.

## 4-1) “책임자” = 친구가 리스크를 다 진다? (정확히 구분해야 함)

여기서 말하는 책임은 보통 2가지가 섞여서 오해가 생깁니다.

### A. 법적 책임(대외 책임)
- **원칙적으로 법적 책임은 서비스 제공 주체(회사/대표)**에게 귀속됩니다.
- 약관/정책/대응/배상 등 최종 책임은 **“개인”이 아니라 “사업 주체”**가 지는 구조입니다.
- 따라서 “마케팅 하는 친구를 PM으로 세운다 = 친구가 법적 책임을 다 진다”는 의미는 **보통은 아닙니다.**

### B. 운영/의사결정 책임(내부 책임)
- PO/PM은 **무엇을 약속할지(정책/문구/범위)**를 결정·관리하는 역할이라,
  - 문제가 터졌을 때 “누가 어떤 기준으로 결정했는가”라는 **내부 책임(프로세스 책임)**은 생깁니다.
- 하지만 이 또한 **역할과 권한을 문서로 정리**하면,
  - 친구는 “마케팅/제품 정책”을,
  - 대표/법무는 “대외 약관/법적 리스크”를,
  - 개발(본인)은 “기술 구현/품질”을
  나눠서 가져갈 수 있습니다.

### C. 대표님께 이렇게 설명하면 깔끔합니다(짧은 지문)
> “PM을 둔다는 건 그 사람이 법적 책임을 떠안는다는 뜻이 아니라,  
> **무엇을 만들고 어떤 정책으로 운영할지**를 한 사람이 책임지고 정리해서  
> 개발이 흔들리지 않게 만드는 역할입니다.  
> 법적 책임은 회사/대표가 지고, PM은 내부 의사결정과 운영 품질을 책임집니다.”

> 참고: 아래 내용은 일반적인 제품/운영 관점 정리이며, **법적 해석은 변호사/노무사 상담이 안전합니다.**

## 5) GoldSilver(금은세상) 프로젝트 파이프라인(현재/권장)

### 현재 개발 파이프라인(로컬 기준)
- 개발: Next.js(App Router) + TypeScript + Tailwind
- 실행: `npm install` → `npm run dev`
- 데이터:
  - 공공데이터포털(금 시세) + 카카오 로컬(주변 매장)
  - **API 키가 없거나 실패 시 더미 데이터로 폴백**
- 상태 저장: 브라우저 `localStorage`(예: 다크모드, 사용자, 알림 등)

### 권장 배포 파이프라인(출시용, 간단 버전)
1. **GitHub**(소스)
2. **CI**(lint/build/test)  
3. **CD**(스테이징 → 프로덕션)  
4. **모니터링**(에러 트래킹/로그/성능)  

예: GitHub Actions + Vercel(또는 Cloudflare Pages)

## 5-1) 대표님이 “어플 어떤식으로 만들어요?”라고 물으면(1분 설명)

### A. 1단계: 웹(Next.js)로 먼저 만든다 (지금 상태)
- 웹으로 만들면 **배포/수정이 빠르고**, 사용자 피드백을 빨리 받을 수 있습니다.
- 핵심 화면/기능(홈·차트·거래·포트폴리오·알림·매장 등)을 먼저 완성해 MVP를 검증합니다.

### B. 2단계: “앱처럼” 쓰게 만든다 (PWA)
- 동일한 웹을 **PWA로 설치형 앱처럼** 쓸 수 있게 합니다.
- 장점: 스토어 심사 없이도 설치 경험 제공, 업데이트 즉시 반영
- 한계: iOS/일부 기능(푸시/생체인증 등)은 제약이 있어 범위 정의가 필요

### C. 3단계: 진짜 앱이 필요하면 패키징(예: Capacitor)
- 같은 웹 코드를 **iOS/Android 앱 껍데기**로 감싸서 스토어 배포가 가능합니다.
- 이때부터는 기기 기능(생체인증, 푸시, 딥링크 등)을 네이티브 플러그인으로 붙입니다.

### D. “실제 돈이 오가는 거래”를 한다면(중요)
- 프론트만으로는 불가하고, 반드시 **백엔드/DB(원장/정산/중복방지)**가 필요합니다.
- 현실적인 구조:
  - 프론트(Next.js) + 백엔드(API) + DB(원장) + 결제/정산(PG/파트너) + 모니터링
- 여기서부터는 “화면”이 아니라 **정합성/보안/운영/법무**가 개발 난이도를 결정합니다.

## 6) GoldSilver(금은세상) 핵심 로직(간단 설명)

### A. 시세 로직
- `fetchGoldPrices()`  
  - 공공데이터포털 API로 금 가격 조회  
  - 응답에서 “1kg 기준” 값을 찾아 **g당 가격으로 환산**  
  - 금 순도(24K/18K/14K) 비율(ratio)로 순도별 가격 생성  
  - 은 가격은 금 대비 비율로 생성(데모) + 순도별 ratio 적용

### B. 차트 로직
- 기간 선택은 “최근 n일”이 아니라 **간격 기준**
  - 1주: 1주 간격(여러 주)
  - 1개월: 1개월 간격(여러 달)
  - 3개월: 3개월 간격(분기 단위)
  - 1년: 1년 간격(여러 년)
- 단위 선택:
  - g ↔ 돈(1돈 = 3.75g) 변환
- 금/은 가격 범위가 달라서, Y축 라벨은 **가격 크기에 따라 자동 포맷**(만/천/원)

### C. 거래/자산 로직(데모)
- 거래 화면:
  - 매수/매도 선택
  - **매도 수량은 보유량 초과 불가**
  - 총액 = 수량 × 단가 (+ 매수 시 부가세 10% 데모)
- 포트폴리오:
  - 총 평가금액/투자원금/손익/손익률 계산

### D. 알림/설정 로직(데모)
- 가격 알림: 목표가(이상/이하) 저장/활성화 토글
- 다크모드: `localStorage` 값으로 초기 로드 시 바로 적용(깜빡임 최소화)
- 고객센터: FAQ/문의 채널 페이지 제공(정책·운영 문구는 출시 전 확정 필요)

## 7) “출시”를 위해 추가로 필요한 것(현실 체크리스트)

### 제품/정책
- 사용자 시나리오별 정책(수수료/세금/단위/정확도/출처/면책)
- 약관/개인정보처리방침/마케팅 동의
- 고객센터 운영 정책(SLA, 응대 시간, 환불/분쟁 프로세스 등)

### 기술
- 로그인/세션/권한(실서비스 수준)
- 백엔드/DB(보유 자산, 거래 내역, 알림 스케줄링, 사용자 설정)
- 알림(Push) 실제 구현(PWA/모바일 패키징 시 FCM/APNs 등)
- 모니터링(Sentry 등), 성능 최적화, 보안 점검

## 8) 대표님께 제안하는 “현실적인 운영 방식”

- **PO/PM(기획·책임자) 1명은 반드시**: 스코프/우선순위/정책/리스크 문구 확정
- 개발은 AI를 적극 활용하되, **릴리즈 체크리스트(품질·보안·법무)**를 운영 프로세스로 만들기
- MVP → 베타 → 정식 런칭으로 단계 분리(각 단계마다 “성공 기준” 명확화)
